HTH 2.0 Tech Stack Assessment

To prepare for the Hills To Home 2.0 project, I led an initiative to establish a modernized, future-proof frontend architecture. The goal was to balance cutting-edge developer experience with the rigorous maturity, stability, and integration requirements of a large-scale enterprise environment. Below is the outcome of that architectural decision matrix.

Application Framework

Selected

Next.js

Chosen for its feature-rich environment, massive enterprise adoption, and out-of-the-box routing. Its server-side API routes provide an excellent integration layer for the broader SAP ecosystem.

Alternatives Evaluated

  • ✕
    Remix: Promising, but the ecosystem was too young and presented a higher risk profile for our scale.
  • ✕
    Bare React (CRA): Lacks built-in SSR and routing; requires too much manual boilerplate maintenance.

State & Data

Selected

RTK + React Query

Redux Toolkit (RTK) handles complex, predictable global state transitions, while React Query abstracts away the heavy lifting of caching, synchronizing, and fetching remote SAP API data.

Alternatives Evaluated

  • ✕
    Zustand / Recoil: Excellent DX, but lacked the battle-tested enterprise maturity and strict architectural guardrails of Redux.
  • ✕
    Context API: High risk of performance bottlenecks due to unnecessary re-renders with highly dynamic data.

Build Tooling

Selected

Webpack + SWC

Selected as the native, zero-configuration default within the Next.js ecosystem. SWC (Rust-based) significantly reduces compilation times without requiring complex custom configurations.

Alternatives Evaluated

  • ✕
    Vite: Incredibly fast and modern, but integrating it outside the standard Next.js compilation flow introduces unnecessary friction.
  • ✕
    Turbopack: Still in active development/beta phases at the time of assessment; too volatile for production standards.
