Classifier — Full Reference

How the LLM agent classifies candidates · search requirements · output schema · validation rules · time-to-resolution tiers

Two Classifier Modes

Regular (Price-Filter) build_regular_prompt()
  • Input: candidates from ScannerAgent — markets already priced ≥80–85c by the crowd
  • Question asked: "Is the apparently-obvious outcome actually near-certain?"
  • Used by: incremental, full, deep, pm-* scan modes
  • Classifier prompt includes: ticker, prices, volume, urgency score, close date, settlement rules, and any volume anomaly
  • Default assumption: the high-priced side is likely correct; look for reasons it isn't
Anomaly (Volume-First) build_anomaly_prompt()
  • Input: candidates from AnomalyScanner — markets at 20–79c with unusual HC capital ≥$10k
  • Question asked: "Is the smart money right — is the market mispriced?"
  • Used by: anomaly and pm-anomaly scan modes
  • Classifier prompt includes: implied HC dollars vs opposite dollars, HC-to-opposite ratio, settlement rules
  • Default assumption: price is below certainty threshold for a reason — look for what the smart money knows
Both modes share the same output schema and the same validate_classification() rules, so the Opportunity Manager handles them identically downstream. The difference is only in what question the agent is asked and what context appears in the candidate prompt.

Mandatory Search Sequence

Minimum 3 searches required. The recency search is hardcoded mandatory — skipping it causes automatic downgrade to LIKELY.

Regular (Price-Filter) classifier search sequence
A
Current real-world status — What is the ground truth right now? Is the event underway, concluded, or still pending? Official sources preferred.
B
Recency search (MANDATORY — cannot skip)
Query: "[topic] news [current month year]" or "[topic] update [current month]"
Catches late-breaking developments that could invalidate an apparently obvious outcome. Missing this forces LIKELY regardless of confidence.
C+
Settlement criteria — How exactly will Kalshi resolve this market? The settlement source URL and rules_primary text are provided. Verify that the real-world outcome cleanly maps to the settlement rule — mismatches are a contradicting signal.
D
Volume anomaly (if flagged) — If the candidate includes a VOLUME ANOMALY block, there is a mandatory additional search: why might someone have large conviction on the opposite (low-probability) side? Unexplained anomalous volume must be listed as a contradicting signal if no explanation is found.
Anomaly classifier search sequence
A
Ground truth status — What is the current real-world status of this event? Is the outcome already determined or still open?
B
Recency search (MANDATORY) — Volume often front-runs public news. Search for recent announcements, policy decisions, or catalysts that could explain why large capital is already positioned on the high-confidence side.
C+
Smart money rationale — Why would sophisticated players put $10k+ on a 20–79c outcome? Could the volume be hedging, market-making, or a known public event (which weakens the signal)? Or is there information asymmetry at play?

Output Schema — All Fields

The agent must output exactly this JSON structure. No additional fields, no markdown wrapping. validate_classification() enforces the rules post-output.

Field Type Description & Requirements
classification string ALL  One of "CERTAIN" "LIKELY" "UNCLEAR". Only CERTAIN classifications proceed to the Opportunity Manager. LIKELY and UNCLEAR are logged but not acted on.
confidence_score number (0–100) CERTAIN  The agent's numeric certainty. Must be ≥ 95 for CERTAIN. Used downstream for annualised edge calculation and backtest confidence bands.
high_confidence_side string ALL  "YES" or "NO" — which side the agent is confident will win. Used by the backtest to compare against the actual settlement result.
reasons array of strings CERTAIN  Plain-language justifications. CERTAIN requires ≥ 3 reasons. Top 3 are shown in the terminal notification. Should be distinct — not restatements of each other.
confirming_signals array of objects CERTAIN  Facts found during research that support the high-confidence outcome. Each object: {"{"}"fact": "...", "source_url": "..."{"}"}. CERTAIN requires ≥ 3. Source URLs are required — "no URL" should not exist for CERTAIN.
contradicting_signals array of objects CERTAIN  Facts that argue against the obvious outcome. Same format as confirming_signals. CERTAIN requires exactly 0 — any contradicting signal auto-triggers downgrade to LIKELY. Cluster pricing inconsistencies and unexplained volume anomalies also count here.
what_would_change_this string ALL  Required for all classifications. The specific scenario or evidence that would flip the outcome. Empty string fails validation. Forces the agent to articulate its falsifiability conditions. Shown in the terminal notification under "What would change this".
settlement_risk string Any scenario where the platform might resolve differently than the real-world outcome — e.g., ambiguous settlement rules, jurisdiction-dependent definitions, timing edge cases. Empty string is acceptable if no settlement risk exists. Shown in the terminal notification.
recent_developments string CERTAIN  The findings from the mandatory recency search. Must be non-empty for CERTAIN — "None found" only if the search genuinely returned nothing. Empty = auto-downgrade to LIKELY (recency search was skipped). Window controlled by KALSHI_RECENCY_DAYS env var (default: 14 days).
searched_for array of strings ALL  List of actual queries performed. All classifications require ≥ 3. The recency query (Search B) must appear here. Used to audit whether the agent actually searched before classifying.

Validation Rules — validate_classification()

Applied in kalshi_cron.py finalize before the opportunity manager processes results. Violations auto-downgrade to LIKELY and are recorded in _validation_errors.

CERTAIN — required conditions
  • confidence_score ≥ 95
  • len(reasons) ≥ 3
  • len(confirming_signals) ≥ 3
  • len(contradicting_signals) == 0
  • recent_developments is non-empty
All classifications — always required
  • what_would_change_this is non-empty
  • len(searched_for) ≥ 3
  • confidence_score is numeric (int or float)
  • reasons is a list
  • confirming_signals is a list
  • contradicting_signals is a list
  • searched_for is a list
Auto-downgrade triggers (CERTAIN → LIKELY)
  • Any contradicting_signals entry
  • Empty recent_developments
  • confidence_score < 95
  • Fewer than 3 reasons or confirming signals
  • Non-numeric confidence_score → set to 0
Devil's Advocate pass (post-classification): for each CERTAIN result, the agent runs a second check asking for the strongest counter-argument. If a coherent argument (2+ substantive sentences with real-world grounding) can be constructed, the classification is downgraded to LIKELY, the argument is added as a contradicting_signal, and the confidence score is lowered. This happens before the JSON is written to classified.json.

Time-to-Resolution Factor new

The DAYS TO CLOSE field from the candidate prompt is used to calibrate how strictly CERTAIN is applied. Near-term markets have limited surface area for surprises; far-horizon markets carry compounding black-swan risk. This is a soft constraint — it guides the agent's reasoning but is not enforced by validate_classification().

Time Remaining Difficulty Rationale Guidance
≤ 7 days Easy Little time for new information, policy reversals, or unexpected events. The real-world situation is essentially locked in. If current reality strongly favors the outcome and no near-term catalyst for reversal exists, CERTAIN is appropriate with normal scrutiny.
8–30 days Standard Normal window. Some risk of new developments but limited. Apply the standard threshold. Require 3 confirming signals. Do not relax or tighten beyond the base rules.
31–90 days Hard Multiple weeks allow for: earnings surprises, regulatory decisions, geopolitical events, organizational changes. Require stronger evidence. Lean toward LIKELY unless there is a structural impossibility. Flag any uncertain assumptions explicitly.
91–365 days Very Hard A full quarter or more. Macro conditions, leadership changes, black swans become material probabilities. Only structural impossibilities should be CERTAIN — e.g., an acquired company that legally cannot IPO, a mathematical upper bound on a statistic. Standard "current trajectory" reasoning insufficient.
> 365 days Almost Never Year-plus horizon. Essentially any outcome is possible given sufficient time. Reserve CERTAIN for logical or mathematical impossibilities only. LIKELY is almost always the ceiling here.

Category-Specific Notes

🗳 Political (elections, legislation, appointments)
  • Far-horizon markets are inherently unpredictable — incumbents lose, bills fail on procedural votes, nominees withdraw
  • Election markets: never CERTAIN more than ~7 days out unless an opponent withdrew, race was cancelled, or only one party filed
  • Appointment markets: confirmation is near-certain only after a Senate vote is scheduled and counted — not just after nomination
  • Legislative markets: floor vote scheduled ≠ passage; even after passage, executive signature is a separate event
  • Watch for: special elections, recalls, resignations, and redistricting
🏢 IPO / Corporate Actions
  • Timelines shift constantly — S-1 filings are withdrawn, postponed, or repriced without warning
  • "By [month]" markets: CERTAIN NO only if: (1) company was acquired, (2) company went bankrupt, (3) explicit public withdrawal with no window to re-file before close, or (4) the month has passed
  • Mergers & acquisitions: "Will X acquire Y by [date]?" — CERTAIN only after regulatory approvals and shareholder votes are complete
  • Settlement risk is high: Kalshi may resolve based on first trading day on exchange, not signing date or announcement
  • Watch for: SPAC alternatives, direct listings (may not count), and dual-class structures that can delay public float
📊 Economic Indicators (CPI, GDP, unemployment, Fed rate)
  • Only near-term prints with known underlying data can be CERTAIN
  • Monthly data releases (CPI, jobs report): can be CERTAIN 1–3 days before release if nowcast models converge and no revisions are pending
  • Fed rate decisions: CERTAIN only after the FOMC statement is published — market pricing of 95%+ is not the same as a decision
  • GDP: subject to multiple revisions; "advance" estimate ≠ final; be cautious about what the settlement criteria specifies (advance vs. second vs. final)
  • Longer-horizon economic markets (annual totals, year-end indicators): almost never CERTAIN
⚽ Sports
  • Only CERTAIN for games happening within days with results verifiable
  • Head-to-head outcomes: CERTAIN only if the game has already been played (live result) or the opponent forfeited/withdrew
  • "Champion by end of season" markets with many games remaining: almost never CERTAIN regardless of standings
  • Player performance markets: injury, lineup, and weather create non-trivial uncertainty even in games days away
  • Settlement risk: check whether Kalshi settles on the final result, including overtime/shootout, or on regulation only
  • Watch for: schedule changes, disqualifications, venue switches, weather cancellations

Cluster Context — Market Grouping Input

Each candidate prompt may include a CLUSTER CONTEXT block listing sibling markets from the same event. This is appended automatically by market_clusterer.py before the classifier receives the candidate.

Cluster context rules for the classifier:
  • Classify the primary market on its own merits first — don't average across siblings
  • If a sibling is priced inconsistently (e.g., "by June" NO@90c vs "by September" NO@80c for a cumulative event), treat the inconsistency as a contradicting signal — it means the crowd is logically inconsistent and the market may be wrong
  • If siblings are all pricing the same direction strongly, this can reinforce a CERTAIN classification — it means multiple independent markets have converged
  • If a [!] PRICING INCONSISTENCY DETECTED block appears, it was flagged by the monotonicity check and must be addressed in reasoning
// Example CLUSTER CONTEXT block (appended to candidate prompt)
CLUSTER CONTEXT — 4 related markets under the same event:
  [NO@91c] Will Databricks IPO by March 2026?
  [NO@87c] Will Databricks IPO by April 2026?
  [NO@82c] Will Databricks IPO by May 2026?

[!] PRICING INCONSISTENCY DETECTED:
  - 'Will Databricks IPO by March 2026?' at 91c > 'Will Databricks IPO by April 2026?' at 87c
    (later deadline should price >= for a cumulative outcome)
