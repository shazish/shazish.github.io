Backtest — Retrospective Precision Scorer

Measures classifier accuracy against actual market resolutions · target ≥ 95% precision on CERTAIN · run manually via python3 kalshi_cron.py backtest

What It Measures

The backtest reads every opportunity logged to logs/opportunities.jsonl since the pipeline began, queries the Kalshi API for each market's resolution status, and computes what fraction of CERTAIN-classified predictions were correct. The target is ≥ 95% precision. Falling below this threshold indicates the classifier is being too aggressive with CERTAIN and the classification prompts need recalibration.

Why precision, not accuracy? In prediction markets we care deeply about false positives — times we were CERTAIN but wrong — because each miss represents money lost on a bet we were confident about. A recall metric (catching every real certainty) matters less than ensuring that the bets we do make are correct. Hence the 95% precision target, not a balanced F1 score.

Resolution Checking

Each entry in opportunities.jsonl has a candidate.ticker. The backtest calls KalshiClient.get_market(ticker) and checks the status and result fields.

API Field Values Backtest treatment
status "open" / "closed" / "settled" / "finalized" Only settled or finalized markets are scored. Others stay as "pending" in the cache and are re-checked on the next backtest run.
result "YES" / "NO" / "" Empty result = market settled without a binary outcome (voided, cancelled). These are excluded from precision scoring but counted separately.
Results are cached in backtests/resolution_cache.json. Once a market status is settled or finalized, it is never re-queried — the cache entry is permanent. Pending markets are re-queried each backtest run at 150ms/request. A market with status: "error" in the cache had an API failure and is excluded from scoring.

Scoring Formula

Scoring is simple binary: did the predicted side match the actual result? No partial credit, no probability weighting. 

# Core scoring logic (simplified from backtest.py)

for entry, resolution in resolved:
    clf = entry["classification"]
    predicted_side = clf["high_confidence_side"].upper()   # "YES" or "NO"
    actual_result  = resolution["result"].upper()            # "YES" or "NO"
    correct = (predicted_side == actual_result) and (actual_result in ("YES", "NO"))

# Precision = correct / total (for CERTAIN pool)
precision = certain_correct / certain_total  # target: >= 0.95
Minimum sample requirement: precision is only considered meaningful at ≥ 10 resolved CERTAIN markets. Below that threshold, the report prints the raw counts but does not flag whether the target is met. The meets_target field in the JSON output is false until both conditions are met.

Segmentation — Breakdowns

Every resolved market is recorded in up to 4 segments, allowing drill-down when overall precision is weak. Only segments with ≥ 3 resolved markets are shown in the terminal report (noise suppression).

Segment key format Dimension Purpose Example values
CERTAIN|category:X Market category Find whether the classifier is over-confident in a specific domain (e.g., IPO markets are harder than political markets) Politics Business Sports Economics Crypto
CERTAIN|horizon:X Days to close at classification time Find whether far-horizon markets are being mis-classified; expect lower precision in 31-90d and 91-365d buckets <=7d 8-30d 31-90d 91-365d >365d
CERTAIN|confidence:X Classifier confidence score Verify that higher confidence_score actually correlates with higher accuracy; if 98-100 and 95-97 have similar precision, the score is poorly calibrated 98-100 95-97 90-94 <90
LIKELY|category:X LIKELY classification by category LIKELY precision is informational only — it is not a target metric. But consistent precision well above 50% in LIKELY may suggest the threshold between LIKELY and CERTAIN could be lowered in some domains. Same as CERTAIN|category

What you see What it means Action
CERTAIN ≥ 95% with n ≥ 10 Classifier is well-calibrated. CERTAIN classifications are trustworthy. No action needed. Continue monitoring as sample grows.
CERTAIN 90–94%, n ≥ 10 Slightly below target. One or two segments are dragging the overall number. Check which horizon or category segment is underperforming. Tighten that specific domain in the classifier prompt's TTR section.
CERTAIN < 90%, n ≥ 10 Classifier is over-confident. Too many CERTAIN calls are wrong. Review miss list. Raise confidence threshold in validate_classification() (currently 95 → consider 97). Add domain-specific language to the TTR notes. Do a devil's advocate pass audit.
n < 10 resolved markets Insufficient data. Report prints counts but does not flag target. Wait for more markets to settle. First batch expected ~June 2026 (IPO-by-June markets).
LIKELY precision near 50% LIKELY is essentially random — which is acceptable. It means the classifier correctly refrained from CERTAIN on genuinely uncertain markets. No action. LIKELY is not a betting signal, only a tracking signal.

Files

logs/opportunities.jsonl
  • Input to the backtest
  • Appended by finalize mode after every scan cycle
  • One JSON record per line
  • Each record: {"{candidate: {...}, classification: {...}, routing: ..., logged_at: ...}"}
  • Grows over time — never truncated
backtests/resolution_cache.json
  • Caches API resolution results to avoid re-querying
  • Keyed by ticker; values: {"{status, result, checked_at}"}
  • Settled entries are permanent — never re-queried
  • Pending entries are re-checked on every backtest run
  • Safe to delete to force a full re-query (costs API calls)
backtests/backtest_YYYYMMDD_HHMMSS.json
  • Output — one file per backtest run
  • Contains: sample_size, pending, certain{"{...}"}, likely{"{...}"}, segments{"{...}"}
  • The certain.misses list has the full miss records
  • certain.meets_target: boolean — true if n≥10 AND precision≥0.95
  • Old results are not deleted; accumulate for trend analysis

When to Expect Results

The pipeline started logging opportunities on May 15, 2026. Most classified markets have long horizons (close dates in late 2026 or 2027). Resolution data will accumulate gradually.

178 markets logged to opportunities.jsonl. No resolved markets yet.
June 2026 — First expected resolutions
"IPO by June 2026" markets begin settling. Expect 10–30 resolved CERTAIN markets. First meaningful precision measurement possible if ≥10 CERTAIN markets settle.
Q3–Q4 2026 — Growing sample
Monthly economic indicator markets, mid-year political and sports markets settle. Segment breakdowns become statistically useful (≥3 per segment).
2027+ — Full retrospective
Year-end markets settle. Long-horizon IPO, political, and macro markets resolve. Enough data for confidence band and horizon tier analysis to be reliable.
Running early: running python3 kalshi_cron.py backtest before any markets settle is safe — it prints "No resolved markets yet" and exits cleanly. It still caches the current API status for all 178 pending markets, which speeds up future runs.

How to Run

Manual run

# Via cron entry point (recommended)
python3 kalshi_cron.py backtest

# Directly
cd ~/.hermes/kalshi-tracker
python3 backtest.py

Output files

# Terminal: human-readable report
# backtests/backtest_20260601_120000.json: full metrics for programmatic use
# backtests/resolution_cache.json: updated with any newly settled markets

Periodic backtest via cron

Add a weekly backtest job by editing cron/jobs.json. Example entry to run every Sunday at 08:00: 

{
  "id": "kalshi-backtest-weekly",
  "cron": "0 8 * * 0",
  "description": "Weekly classifier precision backtest",
  "command": "python3 ~/.hermes/kalshi-tracker/kalshi_cron.py backtest"
}
Polymarket backtest: not yet implemented. The backtest only scores Kalshi markets. Polymarket entries in opportunities.jsonl are identified by platform == "Polymarket" and skipped with status: "unsupported". Implementing Polymarket resolution requires querying the Gamma API's CLOB endpoint for market results.
